# Interfaces

Generated by `cargo run --example planner_experiments -- --write-docs`.

## Stable Core

- `roboclaw_agent::Planner`
- `roboclaw_agent::PlanDecision`
- `roboclaw_skills::SkillCatalog`
- `roboclaw_tools::Tool`
- `roboclaw_sim::RobotBackend`
- `roboclaw_ros2::Ros2Bridge`
- `roboclaw_memory::Memory`
- Skill YAML files under `skills/`
- Gateway/runtime behavior used by examples and ROS2 flow

## Stable Core Shape

- Planning stays stable only at the boundary: choose one skill from the catalog and return a `PlanDecision`.
- Execution stays stable only at the orchestration boundary: execute tool-backed YAML steps, persist memory, and either continue, resume from checkpoint metadata, or request replanning.
- Recovery stays stable as YAML metadata, not as new runtime abstractions. Core fields are `expect`, `max_retries`, `resume_from_step`, `recovery_for`, `resume_original_instruction`, and `supports_checkpoint_resume`.
- Observation handling stays stable as an input boundary, not as one winning reducer. Core accepts reduced context from ROS2/backend layers while reducer and fusion policy stay experimental.
- Provenance and release-governance policies remain outside core even when budget-style variants win repeatedly.
